## 1. Введение

Данный документ описывает план тестирования учебного проекта `barcode_generator`.
Цель тестирования — проверить соответствие приложения требованиям из `Documents/Requirements/Requirements.md`.

Проект представляет собой веб-приложение для генерации кодов типов:
- QR-code;
- Data Matrix;
- EAN-13;
- EAN-8.

Проверяются как функциональные, так и нефункциональные требования, включая позитивные/негативные сценарии и use case-потоки.

## 2. Объект тестирования

### 2.1 Краткое описание

Приложение позволяет пользователю:
- выбрать тип генерируемого кода;
- выбрать тип кодируемой информации (для QR/Data Matrix);
- выбрать формат изображения (`PNG`/`SVG`);
- сгенерировать и скачать результат.

### 2.2 Компоненты

- `src/public/index.html` — структура UI;
- `src/public/styles.css` — оформление UI;
- `src/public/app.js` — клиентская логика валидации/запросов/превью;
- `src/server.js` — API генерации и серверная валидация.

### 2.3 Атрибуты качества (ISO/IEC 25010)

- функциональная пригодность (полнота, корректность);
- надежность (корректная обработка ошибок ввода/сети);
- удобство использования (понятный интерфейс и сообщения ошибок);
- производительность (приемлемое время генерации);
- совместимость (работа в браузере при локальном запуске Node.js).

## 3. Риски

К рискам проекта относятся:
- некорректный ввод (пустые поля, неверные длины EAN, невалидный email/телефон);
- ошибки в контрольной цифре EAN;
- недоступность сервера при тестировании;
- различия поведения в разных браузерах;
- возможные изменения/ограничения сторонних библиотек (`qrcode`, `bwip-js`);
- регрессии после внесения изменений в клиентскую или серверную логику.

## 4. Аспекты тестирования

### 4.1 Функциональные аспекты

Планируется проверить:
- выбор типа кода: `QR`, `Data Matrix`, `EAN`;
- выбор варианта EAN: `EAN-13`, `EAN-8`;
- выбор формата выходного файла: `PNG`, `SVG`;
- поддержку типов данных для QR/Data Matrix:
  - текст/URL,
  - телефон,
  - email,
  - Wi-Fi,
  - vCard;
- генерацию изображения и его скачивание;
- отображение сообщений об ошибках.

### 4.2 Нефункциональные аспекты

Планируется проверить:
- корректность и единообразие UI;
- отзывчивость интерфейса (кнопка, подсказки, блоки полей);
- устойчивость к неверному вводу;
- базовую производительность (типовой ответ до 2 секунд).

### 4.3 Use Case-тестирование

Основные пользовательские потоки:
- UC-01: генерация QR из текста/URL;
- UC-02: генерация Data Matrix;
- UC-03: генерация EAN-13;
- UC-04: генерация EAN-8;
- UC-05: генерация Wi-Fi QR;
- UC-06: генерация vCard QR.

## 5. Подходы к тестированию

Используется ручное тестирование:
- системное (end-to-end через UI);
- интеграционное (взаимодействие UI с API `POST /api/qr`);
- негативное (проверка обработки ошибок);
- регрессионное (повтор критических сценариев после изменений).

Инструменты:
- браузер (Chrome/Edge/Firefox);
- DevTools (Network/Console);
- GitHub (хранение документов, фиксация результатов);
- при необходимости Postman/curl для проверки API.

## 6. Представление результатов

Результаты тестирования фиксируются в документе:
[`Testing/TestResults.md`](./TestResults.md).

Структура тест-кейса:
- ID;
- назначение (описание);
- сценарий (шаги);
- ожидаемый результат;
- фактический результат;
- оценка (`Pass`/`Fail`).

## 7. Выводы

Данный план покрывает ключевые требования проекта `barcode_generator` и задает единый формат проверки качества. Выполнение тестов по этому плану позволяет подтвердить корректность реализации основных функций, выявить дефекты на раннем этапе и предоставить основу для дальнейшего развития проекта.
